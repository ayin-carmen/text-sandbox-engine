专业化文字沙盒引擎架构方案
============================

一、定位与边界
--------------

本方案面向一个长期演化的文字沙盒游戏引擎。引擎应服务于高状态密度、高事件密度、高分支复杂度的叙事型沙盒项目，并支持后续逐步引入地图、角色、关系、物品、任务、社会规则、内容过滤、扩展模块与编辑工具。

当前阶段不预设完整玩法集合。引擎的首要目标是建立稳定的状态管理、命令执行、规则判断、效果应用、内容装载、扩展注册、存档兼容与调试追踪能力。

引擎核心不直接绑定具体题材、具体 UI、具体剧情结构或具体成人内容规则。题材、玩法与内容应通过数据包和模块扩展接入。


二、核心设计原则
----------------

1. 薄核心

   核心层只保留跨玩法通用能力，包括状态存储、命令管线、注册表、规则执行、效果执行、内容加载、存档、日志和错误处理。

2. 数据驱动

   角色、地点、事件、物品、任务、阵营、规则参数与文本内容应尽量外置，避免将内容逻辑硬编码在 Python 类中。

3. 命令驱动

   玩家行为、脚本行为、系统行为统一表达为命令。所有状态变化都应来源于可追踪的命令执行过程。

4. 事务化状态变更

   状态修改应通过明确的效果集合提交，支持验证、追踪、回滚、日志记录和存档一致性。

5. 注册式扩展

   新模块通过注册组件、命令、规则、效果、钩子与内容类型接入引擎，而不是修改核心流程。

6. 确定性执行

   核心游戏循环应保持确定性，便于存档、复现、调试、自动测试和事件路径验证。

7. 可观测性优先

   引擎必须能够回答：命令为何成功、为何失败、哪些条件被检查、哪些效果被执行、哪些状态被修改、哪些事件进入候选池、哪些事件被过滤。

8. 内容安全与内容治理前置

   事件标签、内容分级、角色约束、过滤策略、审计字段应作为内容系统的一部分，而不是后期补丁。


三、总体架构
------------

推荐架构：

    Runtime
      |-- StateStore
      |-- CommandPipeline
      |-- TransactionManager
      |-- Registry
      |-- RuleEngine
      |-- EffectEngine
      |-- ContentRepository
      |-- SceneOrchestrator
      |-- Persistence
      |-- Diagnostics
      |-- ModuleLoader

各部分职责如下：

1. Runtime

   引擎运行时入口，负责初始化核心服务、加载模块、加载内容、创建或恢复世界状态，并向 UI 层提供稳定接口。

2. StateStore

   世界状态的唯一事实来源。负责维护实体、组件、全局变量、历史记录、随机种子、版本信息和运行时元数据。

3. CommandPipeline

   统一处理玩家命令、脚本命令和系统命令。负责命令解析、权限检查、上下文构建、验证、执行和结果返回。

4. TransactionManager

   将效果执行组织为事务。负责状态变更收集、冲突检查、原子提交、变更记录、失败处理和必要的回滚能力。

5. Registry

   统一注册组件类型、命令处理器、规则处理器、效果处理器、事件触发器、内容类型、钩子和验证器。

6. RuleEngine

   负责执行条件判断。规则应保持纯判断属性，原则上不直接修改状态。

7. EffectEngine

   负责执行状态变更。所有效果执行必须通过事务上下文，并生成可追踪的变更记录。

8. ContentRepository

   负责加载、验证、索引和查询外部内容。内容包括实体定义、事件、场景、任务、地点、物品、阵营、日程、文本和本地化资源。

9. SceneOrchestrator

   负责从当前状态中选择可展示内容，包括地点描述、可用行动、候选事件、场景接续、选项可见性与内容过滤。

10. Persistence

   负责存档、读档、自动保存、版本迁移、状态快照和兼容性校验。

11. Diagnostics

   负责调试追踪、命令日志、规则日志、效果日志、内容校验报告、事件候选报告和性能指标。

12. ModuleLoader

   负责模块发现、依赖解析、初始化顺序、注册调用和模块版本管理。


四、核心数据模型
----------------

引擎应采用实体组件模型，而不是为玩家、NPC、地点、物品分别建立高度固定的继承结构。

1. Entity

   表示世界中的可识别对象。玩家、NPC、地点、物品、阵营、任务、路线都可以是实体。

   基础字段：

   - id
   - type
   - components
   - tags
   - metadata

2. Component

   表示实体的一组结构化数据。组件应可独立注册、验证、保存和迁移。

   组件要求：

   - 有唯一类型名
   - 有 schema
   - 有版本号
   - 可序列化
   - 可迁移

3. WorldState

   表示当前完整游戏状态。

   建议包含：

   - schema_version
   - runtime_version
   - seed
   - entities
   - globals
   - flags
   - history
   - cooldowns
   - indexes
   - diagnostics_state

4. Command

   表示一次意图明确的行为。

   建议字段：

   - id
   - type
   - actor
   - target
   - args
   - source
   - timestamp
   - metadata

5. Rule

   表示条件判断。规则可以用于命令验证、事件触发、选项显示、地点准入、任务推进和内容过滤。

   规则要求：

   - 无副作用
   - 输入明确
   - 输出结构化结果
   - 失败原因可追踪

6. Effect

   表示一次状态修改意图。

   效果要求：

   - 通过事务执行
   - 生成变更记录
   - 支持失败报告
   - 支持权限或约束检查

7. Scene

   表示可展示给玩家的一段交互内容。

   建议包含：

   - id
   - priority
   - scope
   - conditions
   - content_tags
   - text
   - choices
   - cooldown
   - repeat_policy


五、命令执行流程
----------------

标准命令流程：

    Receive Command
      -> Normalize
      -> Build Context
      -> Validate Rules
      -> Build Transaction
      -> Apply Effects
      -> Commit State Changes
      -> Emit Runtime Events
      -> Update Indexes
      -> Select Presentation
      -> Return Result

关键要求：

1. 命令执行必须生成 CommandTrace。
2. 规则验证失败时不得产生状态变更。
3. 效果执行失败时事务必须进入明确的失败状态。
4. 所有状态差异必须进入 ChangeSet。
5. 场景选择应在状态提交后进行。
6. 随机结果必须依赖可保存的随机种子。
7. UI 不应直接修改状态，只能提交命令。


六、模块系统
------------

模块是引擎扩展的基本单位。每个模块负责向核心注册能力，不应绕过核心直接控制运行时。

模块可注册内容：

1. Component Schema
2. Command Handler
3. Rule Handler
4. Effect Handler
5. Runtime Hook
6. Content Validator
7. Index Builder
8. Migration
9. Debug Reporter

模块元数据应包含：

1. module_id
2. version
3. dependencies
4. compatible_engine_version
5. provided_components
6. provided_commands
7. provided_rules
8. provided_effects
9. migrations

模块加载流程：

    Discover Modules
      -> Check Compatibility
      -> Resolve Dependencies
      -> Register Schemas
      -> Register Rules and Effects
      -> Register Commands
      -> Register Hooks
      -> Validate Content
      -> Build Indexes

模块边界要求：

1. 模块不得直接修改核心执行流程。
2. 模块不得直接写入 StateStore，必须通过事务和效果。
3. 模块之间不应硬依赖内部实现，应通过注册表和公共查询接口协作。
4. 模块必须提供清晰的内容 schema 和错误报告。


七、内容系统
------------

内容系统应承担加载、校验、索引、查询、过滤和本地化职责。

内容类型可包括：

1. entities
2. locations
3. routes
4. actors
5. items
6. factions
7. events
8. scenes
9. quests
10. schedules
11. text resources
12. localization resources

内容处理流程：

    Load Files
      -> Parse
      -> Schema Validate
      -> Reference Validate
      -> Tag Validate
      -> Build Index
      -> Report Diagnostics

内容系统必须检查：

1. ID 唯一性
2. 引用有效性
3. 组件 schema 合法性
4. 条件和效果是否已注册
5. 内容标签是否合法
6. 冷却与重复策略是否合法
7. 本地化 key 是否存在
8. 存档兼容字段是否明确


八、场景与事件编排
------------------

场景编排层负责决定当前应该向玩家呈现什么，而不是负责底层状态规则。

主要职责：

1. 查询当前上下文。
2. 收集候选场景。
3. 执行场景条件判断。
4. 应用内容过滤策略。
5. 根据优先级、权重、冷却和重复策略排序。
6. 生成可展示选项。
7. 将玩家选择转换为命令或效果集合。

候选筛选维度：

1. 当前地点
2. 当前时间
3. 当前实体状态
4. 当前任务阶段
5. 当前关系状态
6. 当前标签环境
7. 事件冷却
8. 内容标签过滤
9. 剧情优先级
10. 随机权重


九、持久化与版本管理
--------------------

引擎必须从第一版开始考虑存档兼容。

存档应包含：

1. engine_version
2. schema_version
3. enabled_modules
4. module_versions
5. world_state
6. random_state
7. history_summary
8. migration_history

版本管理要求：

1. 每个组件拥有独立 schema version。
2. 每个模块可提供迁移函数。
3. 读档时必须检查模块缺失、版本不兼容和 schema 不匹配。
4. 存档迁移必须生成报告。
5. 调试模式下应支持状态快照对比。


十、诊断与测试
--------------

专业化引擎必须具备可观测性，否则高状态密度项目会很快失控。

Diagnostics 应支持：

1. CommandTrace
2. RuleTrace
3. EffectTrace
4. ChangeSet
5. SceneCandidateReport
6. ContentValidationReport
7. SaveMigrationReport
8. PerformanceReport

测试层级：

1. Registry 测试
2. Rule 单元测试
3. Effect 单元测试
4. CommandPipeline 测试
5. Transaction 测试
6. Content schema 测试
7. Scene selection 测试
8. Save/load roundtrip 测试
9. Migration 测试
10. Deterministic replay 测试


十一、第一阶段交付范围
----------------------

第一阶段只交付引擎底座，不交付完整玩法系统。

必须交付：

1. Runtime
2. StateStore
3. Entity and Component model
4. Registry
5. CommandPipeline
6. TransactionManager
7. RuleEngine
8. EffectEngine
9. ContentRepository
10. SceneOrchestrator
11. Persistence
12. Diagnostics
13. 基础模块加载机制
14. 基础内容 schema 校验
15. 基础自动测试

不纳入第一阶段：

1. 完整地图玩法
2. 完整 NPC 日程
3. 完整衣物系统
4. 完整法律系统
5. 完整经济系统
6. 完整任务系统
7. 内容编辑器
8. Web UI
9. 大规模剧情内容


十二、技术建议
--------------

推荐语言与基础设施：

1. Python 3.11 或更高版本
2. pydantic 或 dataclasses + jsonschema
3. JSON 或 YAML 作为内容格式
4. SQLite 或 JSON 文件作为初期存档载体
5. pytest 作为测试框架
6. rich 作为可选调试输出

设计约束：

1. 核心层不得依赖具体 UI。
2. 内容层不得直接执行任意 Python 代码。
3. 规则和效果必须经过注册表调用。
4. 状态变更必须通过事务。
5. 存档格式必须显式记录版本。
6. 随机行为必须可复现。


十三、风险与控制
----------------

主要风险：

1. 过早抽象
   控制方式：第一阶段只实现真实需要的扩展点，不实现复杂模块生态。

2. 数据格式失控
   控制方式：强制 schema 校验、引用校验和内容诊断报告。

3. 状态修改来源混乱
   控制方式：所有修改必须通过 EffectEngine 和 TransactionManager。

4. 事件难以调试
   控制方式：SceneCandidateReport 必须能显示候选、过滤、失败原因和最终选择原因。

5. 存档无法兼容
   控制方式：组件版本化、模块版本化、迁移报告和 roundtrip 测试。

6. 模块边界膨胀
   控制方式：模块只能注册能力，不直接接管核心流程。


十四、结论
----------

本项目当前不应以“完整系统清单”为起点，而应以“可演化引擎内核”为起点。

第一阶段的核心成果应是一个稳定、可调试、可存档、可扩展的运行时底座。具体玩法系统应在内容制作和需求增长过程中逐步模块化接入。

最终架构目标：

    薄核心
    数据驱动
    命令驱动
    事务化状态变更
    注册式模块扩展
    确定性运行
    完整诊断
    存档版本兼容


十五、未来计划
--------------

未来计划应围绕“先稳定内核，再逐步验证模块”的节奏推进。每个阶段都应形成可运行、可测试、可回滚的交付物，避免在核心尚未稳定时同时展开大量玩法系统。

阶段 0：架构冻结与技术验证

目标：

1. 确认核心架构边界。
2. 确认状态模型、命令模型、规则模型、效果模型的最小接口。
3. 确认内容文件格式与 schema 方案。
4. 确认可复现随机数、日志追踪、存档版本字段的基础设计。

交付物：

1. 架构设计文档。
2. 核心接口草案。
3. 内容 schema 草案。
4. 最小状态样例。
5. 命令执行 trace 样例。

完成标准：

1. 能明确区分核心层、模块层、内容层和 UI 层。
2. 能说明一个命令从输入到状态变更的完整路径。
3. 能说明后续模块如何注册命令、规则、效果和组件。


阶段 1：核心运行时原型

目标：

1. 实现 Runtime。
2. 实现 StateStore。
3. 实现 Entity 与 Component 基础模型。
4. 实现 Registry。
5. 实现 CommandPipeline。
6. 实现 TransactionManager。
7. 实现 RuleEngine 与 EffectEngine。
8. 实现 Diagnostics 基础追踪。

交付物：

1. 可运行的 Python 引擎内核。
2. 基础命令执行流程。
3. 事务化状态变更。
4. 命令日志、规则日志、效果日志。
5. 单元测试。

完成标准：

1. UI 不能直接修改状态。
2. 所有状态修改都必须通过 EffectEngine。
3. 每次命令执行都能输出可读 trace。
4. 规则失败时不会产生状态变更。
5. 相同初始状态、相同命令序列、相同随机种子会得到相同结果。


阶段 2：内容仓库与场景编排

目标：

1. 实现 ContentRepository。
2. 实现内容加载、schema 校验、引用校验和索引。
3. 实现 SceneOrchestrator。
4. 支持事件候选、条件过滤、优先级、冷却、重复策略。
5. 支持选项可见性与选项效果。

交付物：

1. 内容加载器。
2. 内容校验报告。
3. 场景候选报告。
4. 基础事件选择流程。
5. 基础内容包结构。

完成标准：

1. 内容错误能在启动阶段被发现。
2. 未注册规则或效果会被明确报告。
3. 场景为什么出现或为什么被过滤可以被追踪。
4. 内容文件可以在不改核心代码的情况下增加新事件。


阶段 3：最小玩法模块

目标：

只实现验证引擎必要性的最小模块，不追求完整玩法。

建议模块：

1. time
   提供时间组件、时间推进命令、时间判断规则。

2. space
   提供地点组件、连接关系、移动命令、准入规则。

3. narrative
   提供事件、场景、选择、冷却、重复策略。

4. actor
   提供玩家与 NPC 的基础实体、位置、基础属性。

交付物：

1. 最小可玩循环。
2. 基础地点切换。
3. 基础时间推进。
4. 基础 NPC 在场判断。
5. 基础事件触发与选项执行。

完成标准：

1. 玩家可以通过命令改变地点。
2. 移动可以推进时间。
3. 当前状态会影响可见事件。
4. 事件选择会产生可追踪状态变更。
5. 存档读档后结果一致。


阶段 4：持久化与迁移体系

目标：

1. 实现完整存档结构。
2. 实现存档读写。
3. 实现模块版本记录。
4. 实现组件 schema version。
5. 实现基础迁移机制。

交付物：

1. SaveLoad。
2. Save metadata。
3. Migration registry。
4. Save/load roundtrip 测试。
5. Migration report。

完成标准：

1. 存档记录引擎版本、模块版本、组件版本。
2. 缺失模块或版本不兼容时能给出明确错误。
3. 旧版本状态可以通过迁移函数升级。
4. 读档后可继续执行命令并保持确定性。


阶段 5：调试工具与开发者体验

目标：

1. 提供开发期诊断工具。
2. 提供内容校验 CLI。
3. 提供命令回放能力。
4. 提供事件候选分析。
5. 提供状态差异对比。

交付物：

1. debug command trace。
2. content validate 命令。
3. replay 命令。
4. scene candidate report。
5. state diff report。

完成标准：

1. 可以定位某个事件为什么没有触发。
2. 可以定位某个状态字段由哪个命令修改。
3. 可以回放一段命令序列。
4. 可以比较两个状态快照差异。


阶段 6：中世纪沙盒内容验证

目标：

在引擎内核稳定后，用少量中世纪题材内容验证架构，而不是立刻制作大规模内容。

验证范围：

1. 一个小型城镇区域。
2. 少量地点。
3. 少量 NPC。
4. 少量日常事件。
5. 少量关系变化。
6. 少量准入规则。

交付物：

1. 垂直切片内容包。
2. 玩法需求反馈报告。
3. 模块缺口清单。
4. 数据格式修订建议。

完成标准：

1. 内容制作过程中不需要频繁修改核心层。
2. 新需求能通过注册扩展或模块扩展解决。
3. 能识别哪些玩法值得正式模块化。


阶段 7：正式玩法模块扩展

目标：

根据垂直切片暴露出的真实需求，逐步实现正式玩法模块。

候选模块：

1. social
   声望、阵营、传闻、关系。

2. inventory
   物品、容器、金钱、消耗品。

3. clothing
   穿戴、部位、标签、耐久、清洁度。

4. quest
   任务阶段、目标、日志、承诺。

5. schedule
   NPC 日程、地点移动、时间段行为。

6. law
   犯罪、通缉、守卫检查、处罚。

7. economy
   商店、价格、交易、工作收入。

8. content_filter
   内容标签、强度等级、过滤策略、替代事件。

模块化标准：

1. 该功能被多个内容文件重复使用。
2. 该功能需要独立状态。
3. 该功能需要独立规则和效果。
4. 该功能影响多个其他模块。
5. 该功能需要独立测试与调试报告。


阶段 8：工具链与生产化

目标：

当引擎和核心模块稳定后，再建设内容生产工具，而不是过早制作编辑器。

候选工具：

1. 内容校验工具。
2. 事件图谱工具。
3. 任务阶段检查工具。
4. 本地化检查工具。
5. 存档检查工具。
6. 内容标签审计工具。
7. 简易内容编辑器。
8. Web UI 或桌面 UI。

完成标准：

1. 内容作者可以在较少理解 Python 的情况下制作内容。
2. 内容错误能在运行前被发现。
3. 大规模事件库可以被搜索、审计和测试。


长期目标
--------

长期目标不是制作一个固定题材专用脚本播放器，而是形成一个可复用的文字沙盒引擎。

长期能力应包括：

1. 可插拔玩法模块。
2. 可版本化内容包。
3. 可迁移存档。
4. 可复现运行结果。
5. 可追踪状态变化。
6. 可审计内容标签。
7. 可测试事件路径。
8. 可支持 CLI、Web 或其他 UI。
9. 可支持不同题材的内容包。

最终判断标准：

    当新增一个重要玩法时，不需要重写核心。
    当新增一批内容时，不需要修改 Python 逻辑。
    当某个事件行为异常时，可以准确解释原因。
    当存档跨版本读取时，可以明确迁移路径。
    当项目规模扩大时，架构复杂度仍然可控。
