# 阶段 6：中世纪沙盒内容验证报告

本阶段用一个很小的中世纪城镇 `greybrook` 验证当前引擎内核，而不是制作大规模内容。

## 垂直切片范围

地点：

1. `location.west_gate`：西门，玩家初始地点。
2. `location.market_square`：集市广场，核心日常活动地点。
3. `location.chapel_yard`：小教堂院，送货目标地点。
4. `location.lord_keep`：领主塔楼，当前被准入规则阻挡。

NPC：

1. `actor.osric`：西门守卫。
2. `actor.elda`：集市面包师。

事件：

1. 西门守卫询问来路。
2. 集市告示说明塔楼暂停接见。
3. 面包师请求送货。
4. 小教堂院完成送货。

## 验证结果

当前引擎可以在不修改核心执行流程的情况下完成：

1. 玩家通过命令切换地点。
2. 移动自动推进时间。
3. 当前地点、时间、旗标和 NPC 在场状态影响可见场景。
4. 场景选项通过效果修改状态。
5. 被阻挡地点会在命令规则阶段失败，且不会修改状态。
6. 命令回放能生成 trace、候选场景报告和最终状态差异。

## 内容制作反馈

内容作者目前可以通过 JSON 描述小规模地点、NPC、场景、条件和选项效果。

写作体验较顺畅的部分：

1. 事件条件清晰，`rule` + `args` 足够直观。
2. 场景选项效果清晰，`effect` + `args` 适合写简单剧情状态。
3. 场景候选报告能解释为什么事件出现或被过滤。
4. `replay` 命令能快速验证一串内容流程。

仍然别扭的部分：

1. 地点和 NPC 仍放在 world state 里，不是独立 authoring content。
2. 关系变化只能临时用旗标表达，还不能自然表达信任、好感或承诺。
3. 准入规则目前只能表达静态 blocked，不能表达“有许可后可进入”。
4. 场景重复策略字段已经存在，但还没有真正驱动 cooldown/history。

## 结论

阶段 6 验证说明：当前薄核心、命令管线、规则/效果、内容仓库、场景编排和诊断工具已经足够支撑一个小型叙事切片。

下一阶段应进入正式玩法模块扩展，优先补足真实内容制作中暴露出的模块缺口，而不是继续扩大示例内容规模。
