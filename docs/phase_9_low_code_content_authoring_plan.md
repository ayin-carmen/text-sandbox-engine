# 阶段 9：低代码内容制作方案

阶段 9 在阶段 8 可视化编辑器的基础上，把平台从“可视化查看和辅助编辑 JSON”推进到“内容策划可以通过结构化界面完成场景创建、逻辑配置、校验和试玩”。

本阶段聚焦降低现有场景制作流程的使用门槛，不与内容包扩展任务合并，也不在本阶段建设完整插件系统、多人协作平台或通用游戏开发工具。

## 1. 阶段目标

阶段 9 的核心目标是：

> 用户不编写 Python、不记忆内部 ID、不手写 JSON，也能完成常规场景的创建、编辑、保存、校验和运行验证。

本阶段需要形成以下完整闭环：

1. 打开已有工作区。
2. 通过向导创建新场景。
3. 通过引用选择器配置地点、角色、物品、任务和场景。
4. 通过参数表单配置触发条件、选项显示条件和执行效果。
5. 在编辑过程中即时发现并定位问题。
6. 安全保存内容，同时保留 JSON 高级编辑能力。
7. 通过可视化操作启动隔离运行会话。
8. 以人类可读方式查看执行结果和状态变化。

阶段 9 完成后，一名了解剧情设计但不了解项目 JSON 格式的用户，应当能够独立创建一个包含触发条件、两个选项和多个状态效果的场景，并完成校验和试玩。

## 2. 当前基础与主要问题

阶段 8 已经提供：

1. 工作区和内容树。
2. 场景结构化表单。
3. Registry 编辑器元数据。
4. Monaco JSON 源码编辑。
5. 内容关系图。
6. 内容校验和稳定诊断 DTO。
7. 隔离运行会话。
8. Trace、State Diff、Changed By 和场景候选分析。
9. 原子保存、备份和外部修改冲突检测。

当前主要使用障碍包括：

1. 规则和效果参数仍需填写 JSON 数组。
2. 角色、地点、物品、任务和场景 ID 需要手动输入。
3. 新建场景缺少向导，固定默认 ID 容易冲突。
4. 条件、选项和效果缺少完整的删除、复制和排序操作。
5. 选项的 `visible_if` 尚未提供结构化编辑。
6. 校验问题不能准确定位到具体表单字段。
7. 切换场景或页面时缺少完整的未保存状态保护。
8. 运行预览仍要求用户编写命令 JSON。
9. Trace 和状态变化主要以原始 JSON 展示。
10. Registry 元数据只能描述基本参数，无法完整驱动低代码控件。

5. 不为了界面便利破坏现有内容包兼容性。
6. 不引入以字符串拼接脚本为核心的新内容逻辑语言。
7. 不在本阶段建设账号、云存储或公网服务。

## 4. 目标用户流程

### 4.1 创建场景

```text
打开工作区
→ 点击“新建场景”
→ 选择场景模板
→ 输入场景名称
→ 选择所属地点
→ 确认自动生成的场景 ID
→ 创建场景
```

用户不需要手动创建文件、确定文件路径或构造初始 JSON。

### 4.2 配置触发条件

```text
点击“添加条件”
→ 选择“角色在场”
→ 从角色列表选择“艾尔达”
→ 条件配置完成
```

界面保存的底层结构仍然是现有规则引用：

```json
{
  "rule": "actor.is_present",
  "args": ["actor.elda"]
}
```

### 4.3 配置选项效果

```text
添加选项“答应送面包”
→ 添加效果“获得物品”
→ 选择角色“玩家”
→ 选择物品“面包篮”
→ 添加效果“推进任务”
→ 选择任务“面包配送”
→ 填写阶段“已接受”
```

### 4.4 运行预览

```text
启动试玩会话
→ 查看当前角色、地点和可用操作
→ 点击移动或场景操作
→ 点击场景选项
→ 查看人类可读结果
```

原始命令、Trace 和 State Diff 保留在默认折叠的“高级详情”中。

## 5. 新建场景向导

### 5.1 向导字段

1. 场景名称。
2. 内容命名空间。
3. 自动生成的场景 ID。
4. 所属地点。
5. 场景模板。
6. 重复策略。
7. 初始优先级。

ID 生成示例：

```text
命名空间：greybrook
名称：tavern keeper request
结果：scene.greybrook.tavern_keeper_request
```

对于中文名称，用户可以同时填写简短的英文标识；后续若增加统一 slug 转换能力，不应改变已经发布的稳定 ID。

### 5.2 创建前检查

1. ID 是否符合场景命名规则。
2. ID 是否已经存在。
3. 目标文件是否已经存在。
4. 地点引用是否有效。
5. 模板生成结果是否通过文档校验。

### 5.3 首批模板

1. 空白场景。
2. 普通叙事场景。
3. NPC 对话。
4. 接取任务。
5. 交付物品。
6. 到达地点事件。
7. 一次性场景。

模板只负责生成符合现有 schema 的标准场景数据，不引入新的运行时语义。

## 6. Registry 驱动的参数表单

### 6.1 元数据扩展

Registry 参数元数据需要从“参数描述”扩展为“可生成编辑控件的完整描述”。

建议结构：

```json
{
  "kind": "effect",
  "type_id": "inventory.add_item",
  "label": "添加物品",
  "category": "物品",
  "description": "向指定角色的背包添加一个物品",
  "module": "inventory",
  "module_version": "1.0.0",
  "parameters": [
    {
      "name": "actor",
      "label": "角色",
      "data_type": "entity",
      "reference_type": "actor",
      "widget": "reference_select",
      "required": true,
      "default": "actor.player",
      "description": "获得物品的角色"
    },
    {
      "name": "item",
      "label": "物品",
      "data_type": "entity",
      "reference_type": "item",
      "widget": "reference_select",
      "required": true,
      "default": null,
      "description": "要添加到背包的物品"
    }
  ],
  "writes": [
    "entities.*.components.inventory.items"
  ]
}
```

### 6.2 控件映射

| 参数信息 | 默认控件 |
| --- | --- |
| `boolean` | 开关 |
| `integer` | 数字输入 |
| `number` | 数字输入 |
| `string` | 文本输入 |
| 带 `enum` | 下拉选择 |
| `actor` | 角色引用选择器 |
| `location` | 地点引用选择器 |
| `item` | 物品引用选择器 |
| `quest` | 任务引用选择器 |
| `scene` | 场景引用选择器 |
| `flag` | Flag 搜索或创建控件 |

### 6.3 兼容和降级

1. 新字段仅作为编辑器增强信息，运行时不依赖 `label`、`widget` 等字段。
2. 缺少增强元数据时，前端回退到基础文本、数字或布尔控件。
3. 无法安全结构化的未知扩展仍允许通过 JSON 高级模式编辑。
4. 类型切换后，保留名称和类型兼容的参数，并为新参数填入默认值。

## 7. 引用选择器与引用索引

### 7.1 引用类型

第一版统一索引以下对象：

1. `actor`
2. `location`
3. `item`
4. `quest`
5. `scene`
6. `flag`

### 7.2 引用 DTO

```json
{
  "id": "actor.elda",
  "type": "actor",
  "label": "艾尔达",
  "source": "world_state.json",
  "valid": true
}
```

### 7.3 数据来源

第一版从以下位置构建引用索引：

1. 当前 world state 中的实体和全局状态。
2. 内容仓库中的场景 ID。
3. 场景条件和效果中的稳定引用。
4. 现有内容关系图索引。

### 7.4 交互要求

1. 支持按中文名称搜索。
2. 支持按稳定 ID 搜索。
3. 支持按引用类型过滤。
4. 显示引用来源。
5. 对丢失引用显示明确警告。
6. 保留高级手动输入能力。
7. 选择器只负责选择，不在本阶段隐式创建角色、地点、物品或任务。

## 8. 规则、选项与效果编辑器

### 8.1 场景条件编辑器

每条条件显示：

1. 中文名称。
2. 技术 ID。
3. 功能说明。
4. 参数表单。
5. 典型读取状态路径。
6. 删除、复制、上移和下移操作。

阶段 9 继续沿用当前场景条件的“全部满足”语义，不新增任意 AND/OR 嵌套，避免在未定义运行时语义前提供误导性界面。

### 8.2 选项编辑器

每个选项包含：

1. 选项文本。
2. 显示条件 `visible_if`。
3. 执行效果 `effects`。
4. 复制选项。
5. 删除选项。
6. 上移和下移。
7. 折叠与展开。

至少保留一个选项；删除最后一个选项时应阻止操作或自动创建一个空白选项。

### 8.3 效果编辑器

每条效果显示：

1. 中文名称和分类。
2. 技术 ID。
3. 功能说明。
4. 参数表单。
5. 可能写入的状态路径。
6. 删除、复制、上移和下移操作。

界面必须明确提示：

```text
效果按照从上到下的顺序执行。
```

## 9. 即时校验与错误定位

### 9.1 校验时机

1. 文档发生变化后，经过短暂防抖执行单文档校验。
2. 从 JSON 高级模式切回表单时执行校验。
3. 保存前再次执行完整文档校验。
4. 创建模板和复制场景后执行校验。

### 9.2 诊断级别

1. `error`：禁止保存。
2. `warning`：允许保存，但需要明确提示。
3. `info`：仅提供建议。

诊断示例：

```json
{
  "severity": "error",
  "code": "reference.missing_target",
  "message": "找不到物品 item.bread_box",
  "json_path": "$.choices[0].effects[1].args[1]",
  "suggestion": "请选择已有物品，或在高级模式中检查该引用"
}
```

### 9.3 表单定位

点击诊断后，编辑器需要：

1. 切换到结构化表单。
2. 展开对应的条件、选项或效果。
3. 滚动到目标字段。
4. 高亮字段。
5. 在字段旁显示错误和修复建议。

## 10. 草稿状态与双向同步

结构化表单和 JSON 编辑器必须共享同一份场景草稿。

要求：

1. 表单修改立即反映到 JSON。
2. 合法 JSON 修改立即反映到表单。
3. JSON 存在语法错误时保留源码，不覆盖最后一份合法结构化数据。
4. 从 JSON 返回表单时提示无法结构化展示的字段。
5. 切换标签不会丢失未保存修改。
6. 切换场景、刷新工作区或关闭窗口前提示未保存内容。
7. 保存成功后更新 revision，并清除脏状态。
8. 外部修改冲突继续沿用阶段 8 的修订标识机制。

## 11. 可视化运行预览

### 11.1 试玩控制台

运行面板从“命令 JSON 编辑器”升级为“试玩控制台”，默认显示：

1. 当前角色。
2. 当前地点。
3. 当前时间。
4. 当前可触发场景。
5. 当前场景选项。
6. 可前往地点。
7. 重置会话按钮。

前端不得自行推断可执行操作，操作列表必须来自真实引擎或编辑器适配层。

### 11.2 人类可读结果

执行结果默认展示业务摘要：

```text
操作成功：答应送面包

状态变化：
- 玩家获得“面包篮”
- “面包配送”进入“已接受”阶段
- 艾尔达的信任度从 0 提升到 1
- 当前场景已记录为看过
```

### 11.3 高级详情

默认折叠以下信息：

1. 原始命令。
2. Rule Trace。
3. Effect Trace。
4. State Diff。
5. 场景候选报告。
6. Changed By。

高级详情继续复用真实运行时和现有诊断能力，不在前端复制执行逻辑。

## 12. 本地 API 边界

### 12.1 Registry 元数据

复用并扩展：

```text
GET /api/metadata/registry
```

增加中文标签、分类、控件类型、引用类型、枚举、帮助信息和模块版本。

### 12.2 引用元数据

建议增加：

```text
GET /api/metadata/references
GET /api/metadata/references?type=actor
```

返回统一引用列表和来源信息。

### 12.3 场景模板

建议增加：

```text
GET  /api/metadata/scene-templates
POST /api/content/scenes/from-template
```

模板生成结果必须经过真实文档校验，并遵守现有安全写入流程。

### 12.4 单文档校验

复用现有接口：

```text
POST /api/validation/document
```

需要增强：

1. 更准确的 JSON path。
2. 参数索引定位。
3. 稳定严重级别。
4. 可选修复建议。
5. 引用有效性诊断。

### 12.5 运行操作

建议增加：

```text
GET  /api/runtime/sessions/{session_id}/actions
POST /api/runtime/sessions/{session_id}/reset
```

`actions` 返回当前可执行的结构化操作，供试玩控制台生成按钮和选择器。

## 16. 验收标准

阶段 9 完成时必须满足：

1. 用户可以不进入 JSON 页面创建、编辑和保存常规场景。
2. 用户不需要记忆角色、地点、物品、任务和场景 ID。
3. 所有内置规则和效果均能自动生成参数表单。
4. 条件、显示条件、选项和效果支持完整增删复制排序。
5. 新建场景不会因为固定默认 ID 重复而失败。
6. 校验问题可以定位到具体字段。
7. 非法文档不能被误保存。
8. 未保存修改不会因为切换页面或场景静默丢失。
9. 用户可以通过按钮完成移动、查看场景和选择选项。
10. 执行结果默认以中文业务描述展示。
11. JSON 高级模式保持可用。
12. 现有场景格式、运行时和内容包保持兼容。
13. 现有测试继续通过。
14. 桌面版本能够完成完整低代码制作闭环。

## 17. 主要风险与控制措施

### 17.1 元数据和实现不一致

风险：Registry 元数据声明的参数顺序或类型与真实规则、效果实现不一致。

控制：

1. 为每个内置注册项增加元数据一致性测试。
2. 参数表单生成后仍调用真实文档和运行时校验。
3. 不让前端元数据替代运行时验证。

### 17.2 引用索引不完整

风险：当前部分角色、地点和物品只存在于 world state，无法获得稳定显示名称。

控制：

1. 第一版允许回退显示稳定 ID。
2. 保留高级手动输入。
3. 对无法解析的引用显示来源和警告，不静默删除。

### 17.3 表单和 JSON 不一致

风险：在高级模式编辑不受表单支持的字段后，切换视图可能丢失数据。

控制：

1. 使用单一草稿模型。
2. 未知字段原样保留。
3. 无法安全映射时阻止覆盖，并提示用户留在高级模式处理。

### 17.4 界面复杂度重新上升

风险：把全部高级能力同时展示，会让低代码表单变成另一种复杂配置界面。

控制：

1. 使用渐进式展开。
2. 默认只显示中文名称、必要参数和常用操作。
3. 技术 ID、读写路径和 Trace 放入高级详情。
4. 优先通过模板减少重复配置。
